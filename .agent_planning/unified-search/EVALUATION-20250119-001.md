## Current State Analysis

### What Exists

✅ **Conversation Search (FTS5)**
- FTS5 virtual table (`conversations_fts`) with stemming + Unicode support
- Backend endpoint: `/api/conversations/search`
- Frontend component: ConversationSearch.tsx with Cmd+K shortcut
- Supports limit/offset pagination
- Project filtering available
- Query strategy: Multi-term OR logic with FTS5 escaping

✅ **Database Schema**
- 8 tables total (requests, conversations, conversation_messages, extensions, todos, plans, subagent_graph_nodes/edges)
- FTS5 indexes for conversations only
- Basic indexes on common filter columns
- 800+ line storage_sqlite.go implementation

✅ **Frontend Infrastructure**
- React 19 + Vite + TanStack Router
- API client hooks with React Query
- Conversation page with search UI
- Message filtering within conversations

✅ **API Structure**
- V1 endpoint pattern: `/api/*`
- V2 endpoint pattern: `/api/v2/*` with pagination headers
- Consistent request/response handling
- HAR file queue processing pipeline

### What's Missing

❌ **Request Content Search** [HIGH IMPACT]
- No FTS5 table for requests
- Cannot search by: endpoint, method, body content, tool names, response text, headers
- Only filtering by model available
- Users must browse all requests linearly
- **Gap severity**: CRITICAL - requests are primary data type

❌ **Extension/Plugin Search** [MEDIUM IMPACT]
- Basic keyword matching, not FTS5
- Cannot search extension descriptions, metadata, documentation
- Extensions Hub has poor discoverability
- **Gap severity**: MEDIUM - secondary feature

❌ **Todos/Plans Search** [LOW-MEDIUM IMPACT]
- No full-text search capability
- Can only list and paginate
- No ability to search across session data
- **Gap severity**: LOW - tertiary feature

❌ **Unified Search API** [ARCHITECTURAL GAP]
- No cross-data-type search endpoint
- No ability to search across requests + conversations + extensions simultaneously
- All searches are type-specific
- **Gap severity**: MEDIUM - affects UX cohesion

❌ **Search Result Context** [UX GAP]
- Conversation matches only return match count, not matching messages
- No highlighting of matches in results
- Cannot drill down into what matched
- **Gap severity**: MEDIUM - affects usability

❌ **Search Analytics** [FEATURE GAP]
- No tracking of search queries
- Cannot analyze user search patterns
- **Gap severity**: LOW - analytics feature

### Ambiguities & Unknowns

#### Architectural Questions

**Q1: Request Body Indexing Strategy**
- *Unknown*: Should requests index full body text, or only specific fields?
- *Impact*: Affects search quality and performance
- *Options*:
  - A) Index all body content (comprehensive but potentially noisy)
  - B) Index specific fields like `model`, `tools_used`, `response.summary` (focused)
  - C) Hybrid: Index structured fields + optional full body search
- *Recommendation*: Option B - focused indexing by specific fields reduces noise and improves relevance

**Q2: Unified Search Scope**
- *Unknown*: Should unified search include ALL data types or only primary ones?
- *Impact*: Affects API design and result handling
- *Options*:
  - A) Unified search includes: requests + conversations + extensions (comprehensive)
  - B) Unified search includes only requests + conversations (core data types)
  - C) No unified search initially, add after per-type search works (incremental)
- *Recommendation*: Option B - cover core data types, extensions can be added later

**Q3: Search Result Context**
- *Unknown*: How much context to include in search results (match count vs. snippets vs. full matches)?
- *Impact*: Affects API payload size and frontend display
- *Options*:
  - A) Match count only (current conversation approach, minimal payload)
  - B) Match count + snippet preview (medium payload, better UX)
  - C) Full matching content (large payload, best UX but slower)
- *Recommendation*: Option B - snippets provide context without large payloads

**Q4: Backwards Compatibility**
- *Unknown*: Should new search endpoints be V2-only or update existing endpoints?
- *Impact*: Affects client adoption and migration path
- *Options*:
  - A) Keep V1 search endpoints unchanged, add V2 variants
  - B) Update V1 endpoints to include new search types (breaking change for clients)
  - C) Gradual migration: V1 delegates to V2 internally
- *Recommendation*: Option A - preserve V1 compatibility, add V2 variants for new searches

**Q5: Text Extraction**
- *Unknown*: How to normalize and extract text from complex nested structures (tool use, tool results)?
- *Impact*: Affects search quality and consistency
- *Options*:
  - A) Simple flat text extraction (fast, may miss context)
  - B) Structured extraction with field markers (better indexing, more complex)
  - C) Both: index both simple and field-marked versions
- *Recommendation*: Option A initially - simple flat text extraction, optimize later if needed
